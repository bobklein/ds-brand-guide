We are not a dev shop. We are not a consultancy. We are scientists — disciplined, evidence-driven, and unafraid of complexity. We work alongside our clients as partners, not vendors. We give them permission to build and leverage software in their own worlds, to solve problems of their own.

3. Methodology

Our four-phase framework is the backbone of how we work and how we talk about our work.

01

Discover

"What should we do?"

Research, opportunity mapping, prioritization. We identify the highest-ROI problem to solve.

02

Experiment

"Does it actually work?"

Working prototypes and MVPs tested on real data, real workflows, real users. Proof before commitment.

03

Engineer

"Make it real."

Sprint-based production engineering. Architecture for scale, security, and integration.

04

Optimize

"Make it better."

Ongoing support, monitoring, continuous improvement until KPIs are hit and beyond.

The Principle

Evidence earns the next investment. Every phase produces proof — a validated hypothesis, working software, measurable outcomes — that justifies moving forward.

Retired Framework

Never use the legacy "Direct, Discover, Design, Develop, Deploy" framework. That is retired. The canonical phrasing is Discover. Experiment. Engineer. Optimize.

Confident, not corporate

Evidence-first, not aspirational

We are partners, not vendors. We use "we" and speak to specific buyer concerns. We acknowledge complexity honestly.

8. Imagery Standards

Every image on the site must earn its place. Images are not decoration — they are visual evidence.

Executive Tone

Calm, deliberate, structured, professional, accountable. If it feels flashy or dramatic — it fails.

One Image, One Job

A product we built, an argument we make, a capability we offer, or a result we delivered.

No Visual Slop

No glowing arrows, neon gradients, 3D effects, hexagon patterns, floating icons, or "futuristic" anything.

Before publishing any screenshot, check for: real patient names (PHI), unapproved internal system names, confidential labels or watermarks, anything marked "draft" or "internal only." When in doubt, blur it or replace with test data.

9. Blog Hero Image System

Every blog hero is a 2-second argument for why Digital Scientists is the right partner.

Core Principle: Argument Over Topic

Topic = what the article is about. Argument = what the article concludes. An image that illustrates the topic is generic. An image that illustrates the argument is specific.

Test: If the image could belong to a different article on the same topic, it fails.

Four Composition Types

Composition A

Partnership Scene

Three-quarter overhead of two professionals at a conference table. Best for collaboration, governance articles. Focal artifact on the table.

Composition B

Decision Desk (Top-Down)

Bird's-eye view of desk with documents. Best for frameworks, evaluations. Central document carries the argument.

Composition C

Lifecycle View

Horizontal progression with checkpoints. Best for method, process maturity, delivery governance articles.

Composition D

Minimal Boardroom

One presentation screen with 2-3 status elements. Best for executive alignment. No dashboards.

Artifact Layer System

Level 1: Focal (Required)

The single most prominent element. Communicates the article's argument. Risk assessment with R/A/G indicators, decision framework with one option highlighted.

Level 2: Context (2-3)

Supporting elements that tell how the conclusion was reached. Workflow with gaps, comparison with strikethroughs, architecture sketch.

Level 3: Environmental (1-2)

Small objects implying professional setting. Coffee cup, pen, notepad. Never compete for attention.

12. How the Site Works

The website is a collection of HTML files in a GitHub repository. All site files live in the public/ directory. When someone merges a change into main, Netlify publishes it within 30 seconds.

There is no CMS, no WordPress admin, no login. The files are the site.

You do not need to know HTML or GitHub to participate. Your job is to produce the content and assets. A technical implementer (or Claude Code) handles the rest.

14. Publishing Workflows

Blog Post: End to End

Phase 1: Create

Outside GitHub

  1. Pick topic and keywords
  2. Draft in Google Docs (use Claude to help)
  3. Internal review and approval
  4. Prepare hero image (1440px wide min)
  5. Organize in shared Drive folder

Phase 2: Build

Claude Code

  1. Open Claude Code, provide content + image
  2. Claude scaffolds page, processes images, sets SEO
  3. Creates branch + Pull Request
  4. Netlify generates preview link

Phase 3: Publish

Review & Merge

  1. Click Netlify preview link
  2. Check content, images, mobile
  3. Approve + Merge in GitHub
  4. Live in 30 seconds

Case Study: Key Differences

  • 1.Get client permission first — can we name them? Show screenshots? Use their logo? Quote someone? Get it in writing.
  • 2.Sanitize all screenshots — no PHI, no confidential data, no unapproved system names.
  • 3.Allow 1-2 weeks for asset gathering — this is always the bottleneck.
  • 4.Extra review scrutiny — Reviewer + Project Lead check metrics, screenshots, attribution.

Landing Page: Key Differences

  • 1.Define the page's purpose, audience, and traffic source
  • 2.Write a content brief (headline, sections, CTA, keywords)
  • 3.Wireframe if non-standard layout
  • 4.Reference an existing page for layout patterns when telling Claude Code what to build

16. Review Process

Fast Lane

Typo fixes, minor text edits, image swaps

Push directly to main. No PR needed.

Review Lane

New pages, posts, design changes, multi-file changes

Pull Request → Netlify preview → Approve → Merge

For Reviewers

  1. Click the Netlify preview link in the Pull Request description.
  2. Browse the new or changed pages. Check content, images, links, mobile layout.
  3. If it looks good: Click "Approve" → "Merge pull request" → "Confirm merge." Done.
  4. If something needs fixing: Leave a comment. The implementer fixes it, the preview updates, you review again.

You never need to look at code. You are reviewing the preview — the actual website.

18. SEO Guidelines

Target Keywords

Primary

HIPAA compliant software development, healthcare AI development, healthcare app development

Secondary

PHI protection, healthcare platform security, MDS optimization software

Long-tail

healthcare app security, HIPAA app development partner, AI for revenue cycle management

On-Page Checklist

  • ✓<title> under 60 chars, keyword near front
  • ✓meta description 150-160 chars
  • ✓OG title + description mirror meta
  • ✓Canonical URL to production domain
  • ✓JSON-LD schema (Organization, Service, FAQ, Article)
  • ✓H1 includes or relates to primary keyword
  • ✓Internal links to 2-3 related pages

Content Strategy Principles

  • Lead with the problem the buyer has, not the solution we sell
  • Use specific numbers and outcomes over vague claims
  • Every page should have a clear CTA
  • Blog posts target long-tail keywords; landing pages target primary keywords
